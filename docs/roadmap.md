# Ari ‚Äî Roadmap

> Generated by /onboard. Check off phases as they ship. Each phase is a vertical
> slice (schema ‚Üí repo ‚Üí service ‚Üí controller ‚Üí route ‚Üí tests) and ends
> deployable and demoable. Source of truth for scope: [prd/ari-phase1.md](../prd/ari-phase1.md).

## What we're building

Ari is a quick contact-capture app for networking events ‚Äî open it on your phone,
capture a person (name, photo, a couple of tags, a one-line note) in under 15
seconds, and it remembers *where* you met them. ¬∑ **Users:** the active networker
(founder, salesperson, DevRel, recruiter) ‚Äî solo, personal contact lists.
**Core resources:** `events`, `tags`, `contacts` (+ contact photos, contact tags,
event quick-tags). **First journey:** at an event, open the app ‚Üí land on the
capture screen ‚Üí snap a photo, tap a tag, save ‚Üí contact stored with location.

All data stays scoped by **both** `organizationId` and `userId` (the template's
org scoping is kept so team sharing can be added later without re-modelling
ownership ‚Äî see PRD ¬ß4). Every repository query filters on the org derived from
the Clerk session ‚Äî never a client value (AGENTS.md, "Architecture rules").

## Reference pattern

`events` (built in Phase 1) is the complete vertical slice ‚Äî copy it for every
resource below. It replaced the starter's throwaway `items` example. The recipe
in **[docs/workshop.md ¬ß2](workshop.md)** still applies verbatim; just read
`events` wherever it says `items`. The wiring touch-points each phase repeats:

- `workers/api/db/schema/<x>.ts` ‚Üí export from `schema/index.ts` ‚Üí `pnpm db:generate && pnpm db:migrate:local`
- `repositories/<x>-repo.ts` (only place Drizzle runs; every query scoped by `org_id`)
- `services/<x>-service.ts` (business rules, domain errors) ‚Üí register in `services/index.ts`
- `controllers/<x>-controller.ts` (zod-validated Hono router) ‚Üí register in `workers/api/index.ts` (authed group)
- `app/routes/app/<x>-*.tsx` (loader/action via `app/lib/api-client.server.ts`) ‚Üí add to `app/routes.ts`
- Tests mirroring all three layers in `tests/` ‚Äî `pnpm test` stays green
- Authorization rules go in `app/lib/capabilities.ts`; limits in `app/lib/plans.ts`

### [x] Phase 1 ‚Äî Events (replaced `items`)  *(done 2026-06-12)*
**Goal:** sign in ‚Üí create an event (name, optional date/venue/notes) ‚Üí see it in
a list ‚Üí activate it; exactly **one active event per user** at a time.
**Slice:** schema `events` ‚Üí `events-repo` ‚Üí `events-service` ‚Üí `events-controller`
‚Üí `app/routes/app/events-list.tsx` (replaced the items index route).
**Shipped:**
- Replaced the `items` slice with `events` end-to-end; removed the items example.
- Schema `events`: `id, orgId, userId, name, date, venue, notes, status('draft'|'active'|'archived'), createdAt, updatedAt`,
  with an `(orgId, userId, status)` index. Reset to a single clean migration.
- One-active-per-user enforced in the **service** (`activate`/`archive`), with the
  deactivate+activate done atomically via `db.batch()` (D1 has no interactive
  transaction). Existence is validated before activating so a bad id never
  archives the live event. Per-user scoping on every repo query.
- Plan gate caps **open** (non-archived) events via `getPlan(plan).maxItems`.
- UI: status badges, activate/archive/delete per card, new-event dialog. Nav + the
  `/api/events` controller wired in; `app/routes.ts` index points at events.
- Tests: 8 added (repo one-active invariant + cross-user isolation, service guards,
  controller routes/validation) ‚Äî **69/69 green**; typecheck + production build pass.
**Done:** create ‚Üí activate ‚Üí active event badged; activating another archives the
previous; archived events still listed; `pnpm test` green; builds & deploys.

### [x] Phase 2 ‚Äî Tags  *(done 2026-06-12)*
**Goal:** create, rename, and delete tags that are **global to the user** and
reusable across events (PRD F1.2).
**Slice:** schema `tags` ‚Üí `tags-repo` ‚Üí `tags-service` ‚Üí `tags-controller` ‚Üí
`app/routes/app/tags.tsx`.
**Shipped:**
- Schema `tags`: `id, orgId, userId, name, createdAt`, **unique per user**
  (`uniqueIndex(orgId, userId, name)`); names trimmed, ‚â§ 30 chars (zod at the edge).
- Uniqueness enforced in the **service** with a new typed `ConflictError`
  (‚Üí HTTP 409, `code: "conflict"`), the DB unique index as the backstop; rename
  allows keeping the same name and rejects clashes with other tags.
- `/api/tags` CRUD (create/list/rename/delete); delete is a plain delete for now
  (Phase 3 will detach from contacts and report the affected count).
- UI: add-tag form, inline per-row rename + delete, duplicate-name toast.
- Tests: 19 added (repo DB-level uniqueness + cross-user isolation, service
  conflict/not-found, controller 409 + validation) ‚Äî **88/88 green**; typecheck +
  build pass. Migration `0001_happy_beast` (additive).
**Done:** build a reusable tag library; duplicates rejected with a clear message;
`pnpm test` green; builds & deploys.

### [ ] Phase 3 ‚Äî Contacts (capture + review CRUD)
**Goal:** the core journey ‚Äî under an active event, capture a contact (name, note,
tags, location) and see/edit it. Name is *softly* required: an empty name saves as
"Unknown" (PRD F2.4).
**Slice:** schema `contacts` + join `contact_tags` ‚Üí `contacts-repo` ‚Üí service ‚Üí
controller ‚Üí `app/routes/app/contacts` (list under an event + detail/edit).
**Build:**
- `contacts`: `id, orgId, userId, eventId, name, note, latitude, longitude, accuracy, capturedAt, createdAt, updatedAt`.
  `contact_tags`: `(contactId, tagId)` ‚Äî repo joins to hydrate tags.
- Service: create accepts a **client-generated id** and is idempotent on it (PRD ¬ß8
  ‚Äî makes the future offline stretch a bolt-on, not a rewrite); empty name ‚Üí "Unknown".
- Loader/action capture **geolocation** client-side at save (browser Geolocation
  API, ~5 s timeout, high-accuracy off); denied/unavailable ‚Üí save without coords,
  never blocked (PRD F2.7). Store `accuracy` so the map can render honestly later.
- List newest-first with photo-placeholder, name, tags, note preview, capture time
  (PRD F3.1); detail page edits all fields, add/remove tags, delete w/ confirm (F3.2).
- Tests: repo (org+event scoping, tag hydration), service (idempotent create,
  Unknown fallback), controller (zod, geo optional).
**Done when:** open an event ‚Üí add a contact with tags + a note ‚Üí it appears
newest-first; editing and deleting work; coords saved when permission granted;
`pnpm test` green; deploys.

### [ ] Phase 4 ‚Äî Contact photos (R2)
**Goal:** a contact can carry **multiple** photos (face + badge), private and
access-controlled (PRD F2.3, ¬ß7).
**Slice:** install `r2-uploads` skill ‚Üí schema `contact_photos` ‚Üí repo ‚Üí service
‚Üí upload + authenticated-serve controller routes ‚Üí photo UI on capture/detail.
**Build:**
- `pnpm install-skill r2-uploads`; create the bucket
  (`npx wrangler r2 bucket create ari-uploads`), uncomment the R2 binding in
  `wrangler.jsonc`, add `R2_PUBLIC_BASE_URL` per the skill's checklist. Wire the
  adapter into `createServices` (services/index.ts).
- `contact_photos`: `id, contactId, r2Key, width, height, byteSize, createdAt`.
- Client-side downscale/compress before upload (target ‚â§ ~500 KB/photo ‚Äî kind to
  conference Wi-Fi). Serve through an **authenticated worker route**, never a
  public bucket; URLs not guessable or long-lived.
- Deleting a contact (and removing a photo) **deletes the R2 object**, not just the
  row ‚Äî run the R2 delete in `waitUntil(...)` (PRD ¬ß7; AGENTS.md side-effects rule).
- Tests: photo repo, upload/serve auth (403 cross-org/cross-user), delete cleanup.
**Done when:** add two photos to a contact, reload, both render via authed URLs;
deleting the contact removes the objects; `pnpm test` green; deploys.

### [ ] Phase 5 ‚Äî Event mode + PWA
**Goal:** the 10-second, one-handed mobile capture experience, plus installability
(PRD ¬ß5.2, ¬ß5.4) ‚Äî the product's reason to exist.
**Slice:** event quick-tags curation (`event_quick_tags`) ‚Üí mobile capture route +
auto-detection ‚Üí PWA manifest + service worker.
**Build:**
- `event_quick_tags`: `(eventId, tagId, position)` ‚Äî an ordered subset (cap ~8) shown
  as one-tap chips. Add curation UI to the event page (F1.3) and an "add tag on the
  fly" affordance that creates a tag (Phase 2) **and** appends it here (F2.5).
- Capture route: on a mobile viewport with an active event, land directly on the
  capture screen ‚Äî camera/photo button, name, quick-tag chips, note, save, top-to-
  bottom, save reachable one-handed (F2.2). Desktop ‚Üí dashboard with an active-event
  banner. Provide "Exit event mode" + a user-overridable "don't auto-enter"
  preference so detection failures aren't trapping (PRD ¬ß6).
- After save: toast + new contact at top of a recent-captures strip; form resets;
  budget < 15 s (F2.8‚Äì2.9). "Start an event" one-step shortcut when none active (F2.10).
- PWA: `public/manifest.webmanifest` (name, icons, standalone, theme color) + a
  service worker precaching the app shell; offline indicator + save-failure keeps
  form state (F4.1‚Äì4.3). *(True offline capture stays a stretch ‚Äî PRD ¬ß8.)*
**Done when:** install the PWA on a phone, open it with an active event, capture a
contact in well under 15 s with a photo + a quick tag; `pnpm test` green; deploys.

### [ ] Phase 6 ‚Äî Global search & filters
**Goal:** find any contact ever captured (PRD F3.3, F3.1 filters).
**Slice:** repo search query ‚Üí service ‚Üí controller ‚Üí a global search route + tag
filters on the event list.
**Build:**
- D1 `LIKE`-based search over contact `name` + `note`, **scoped to the user/org**,
  with optional tag and event filters; results show which event each came from.
- Filter the event contact list by tag(s); sort by time or name.
- `LIKE` is sufficient at phase-1 scale (hundreds‚Äìlow thousands/user); revisit FTS
  only if it proves slow (PRD ¬ß6).
- Tests: search scoping (never leaks another user's contacts), filter combinations.
**Done when:** one search box finds contacts by name or note across all events;
tag filters work; `pnpm test` green; deploys.

### [ ] Phase 7 ‚Äî Map view
**Goal:** see contacts on a map of where you met them (PRD F3.4‚ÄìF3.5).
**Slice:** client-side map (MapLibre GL or Leaflet + OSM tiles ‚Äî no Google key) on
a per-event map route, plus an inline mini-map on contact detail.
**Build:**
- Per-event map plotting each contact at its coords; pins cluster when overlapping
  (venues stack); tap a pin ‚Üí contact card. Contacts **without** coords are listed
  below the map, not lost.
- Contact detail shows a small inline map when coords exist; render honestly using
  the stored `accuracy`.
- No new API key in phase 1.
**Done when:** an event's map shows captured contacts, clustered, with a fallback
list for coord-less ones; contact detail shows its location; deploys.

### [ ] Phase 8 ‚Äî Production launch
**Goal:** move off the development Clerk instance and harden for real use.
**Build:**
- Create a Clerk **production** instance (orgs enabled); update the
  `CLERK_SECRET_KEY` / `CLERK_PUBLISHABLE_KEY` worker secrets and
  `.env.local`'s `VITE_CLERK_PUBLISHABLE_KEY`; redeploy.
- (Optional) wire the Clerk ‚Üí D1 sync webhook: set `CLERK_WEBHOOK_SECRET` and point
  the Clerk endpoint at `https://ari.shanthis.workers.dev/api/integrations/clerk`.
- Instrument the success metrics (PRD ¬ß10): capture-time, photo/tag/coord coverage,
  72-hour return-to-review.
- Pass on perf/PWA (Lighthouse), and confirm photo privacy + geolocation opt-in copy.
**Done when:** production Clerk users can sign up and capture; metrics flow; the PWA
installs cleanly; `pnpm test` green.
